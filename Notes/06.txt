# Day 6 Notes — Data Validation Framework (Trusting the Data)

Day 6 is one of the most important days of the entire project.

Why?

Because until Day 5 we had built a system that could:

```text
Read Data
Log Events
Retry Failures
```

But we still had no guarantee that the data itself was correct.

Day 6 solves:

```text
Data Quality
```

---

# What Was The Goal Of Day 6?

The goal was to ensure:

```text
Only valid data moves through the ETL pipeline.
```

Before Day 6:

```python
df = read_file("employees.csv")
```

File loaded successfully.

Pipeline continues.

But:

```text
File Loaded Successfully
≠
Data Is Correct
```

This is one of the biggest misconceptions in Data Engineering.

---

# The Problem We Are Solving

Imagine:

```csv
employee_id,name,salary
1,John,50000
2,,70000
2,Alice,90000
```

File loads successfully.

But data contains:

```text
Missing name
Duplicate employee_id
```

If this reaches:

```text
Transform Layer
Database Layer
Analytics Layer
```

bad data spreads everywhere.

---

# The Concept Behind Day 6

Day 6 introduces:

```text
Data Validation Layer
```

Architecture becomes:

```text

 ↓
Validate
 ↓
Transform
 ↓
Load
```

Instead of:

```text

 ↓

 ↓
Load
```

---

# Most Important Concept Learned

## GIGO

```text
Garbage In
Garbage Out
```

Meaning:

If bad data enters the system:

```text
Bad Reports
Bad Dashboards
Bad Decisions
```

will come out.

No amount of engineering can fix fundamentally bad input data.

---

# Why Validation Exists

Validation acts as a checkpoint.

Think of it as:

```text
Airport Security
```

Before entering:

```text
Flight
```

you must pass:

```text
Validation
```

Bad data gets stopped immediately.

---

We created:

```python

```

This became the main public API.

Instead of:

```python

validate_schema()
```

being called everywhere,

users simply call:

```python

    config,
)
```

Bad:


Every pipeline repeats this.

# Validation Type 4

## Schema Validation

We implemented:

```python
validate_schema(...)
```

---

# Problem

Expected:

```text
employee_id → Int64
salary → Int64
```

Received:

```text
employee_id → String
salary → String
```

---

# Why This Matters

Many downstream operations depend on data types.

Example:

```python
salary * 1.1
```

Works on:

```text
Int64
```

Fails on:

```text
String
```

---

# ValidationConfig

We created:

```python
ValidationConfig
```

Example:

```python
ValidationConfig(
    required_columns=[
        "employee_id",
        "name",
    ],
    non_nullable_columns=[
        "employee_id",
    ],
    unique_columns=[
        "employee_id",
    ],
    expected_schema={
        "employee_id": "Int64",
        "name": "String",
    },
)
```

---

# Why Configuration Matters

Instead of changing code:

```python
if column == ...
```

users configure rules.

Much more flexible.

---

# Important Engineering Lesson

Understand the difference:

## Transient Errors

Temporary.

Examples:

```text
Network Timeout
API Failure
Connection Reset
```

Retry helps.

---

## Deterministic Errors

Permanent.

Examples:

```text
Missing Column
Duplicate ID
Null Value
```

Retry useless.

# Challenges We Encountered

---

# Error 1

```text
schema field conflict
```

---

## Why?

We initially created:

```python
schema: dict[str, str]
```

inside:

```python
ValidationConfig
```

But Pydantic already has:

```python
schema()
```

method.

Mypy interpreted:

```python
config.schema
```

as:

```python
Callable
```

instead of:

```python
dict
```

---

## Error

```text
Argument has incompatible type
Callable[...] expected dict[str, str]
```

---

## Fix

Renamed:

```python
schema
```

to:

```python
expected_schema
```

---

## Lesson

Avoid naming fields that collide with framework internals.

Examples:

```text
schema
json
dict
copy
fields
```

can cause problems.

---

# Error 2

```text
ValidationConfig updates required
```

After renaming:

```python
expected_schema
```

we also updated:

```python
validator.py
scratch_validator.py
```

to use:

```python
config.expected_schema
```

---

## Lesson

Refactoring often requires updating all dependent code.

# Interview Questions You Can Now Answer

### What is GIGO?

Garbage In, Garbage Out. Poor quality input inevitably leads to poor quality output.

---

### Why validate data immediately after reading?

To fail fast and prevent bad data from spreading downstream.

---

### What is a data contract?

A formal agreement describing expected schema, types, and business rules for data.

---

### Why shouldn't validation errors be retried?

Because they are deterministic data issues, not transient infrastructure issues.

---

### Why separate validation rules into configuration?

To make validation reusable and adaptable across different datasets without changing code.
