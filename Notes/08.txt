# What Was The Goal Of Day 8?

The goal was to build the:

```text
Load Layer
```

which persists transformed data into durable storage.

Before Day 8:

```python
df = read_file(...)

validate_dataframe(...)

df = transform(...)
```

Everything existed only in memory.

If the program stopped:

```text
Memory Cleared
Data Lost
```

After Day 8:

```python
write_file(
    df,
    "employees.parquet",
)
```

Data becomes permanent.

---

# Why Loading Is Important?

In ETL, the final goal is not reading data.

The goal is:

```text

```

so it can be consumed by:

* Dashboards
* Machine Learning
* Analytics
* Databases
* Data Warehouses

Without loading:

```text
ETL = Incomplete
```

Without abstraction:


Bad.

```text

.json
.parquet
```

```python

```

Consistency is good API design.

# Engineering Principle Learned

Users interact with:

```python

```

This is called:

```text
Encapsulation
```

One of the core OOP principles.

# Why Parquet Is Important

Parquet is the standard storage format for modern analytics.

Features:

```text
Columnar
Compressed
Fast
Efficient
```

Used by:

```text
Spark
Snowflake
Databricks
Athena
BigQuery
```

Learning Parquet directly helps Data Engineering interviews.

---

# What Is Columnar Storage?

Traditional CSV:

```text
Row-based
```

Example:

```text
ID Name Salary
1  John 50000
2  Alice 70000
```

Parquet stores data by columns:

```text
ID
1
2

Name
John
Alice

Salary
50000
70000
```

This makes analytics significantly faster.

---

# Real Example

Suppose dashboard only needs:

```text
salary
```

CSV reads:

```text
All Columns
```

Parquet reads:

```text
Only Salary Column
```

Huge performance benefit.

# Exception Handling

We created:

```python
FileWriteError
```

---

# Why?

Writing failures are different from reading failures.

Examples:

```text
Disk Full
Permission Denied
Invalid Path
```

These are write-specific issues.

---

# Why Not Use Generic Exception?

Bad:

```python
Exception
```

Provides little information.

---

Better:

```python
FileWriteError
```

Specific.

Meaningful.

Easy to debug.

---

# Overwrite Protection

We implemented:

```python
overwrite=False
```

by default.

---

# Why?

Imagine:

```python
write_file(
    df,
    "employees.csv",
)
```

File already exists.

Without protection:

```text
Data Lost
```

---

# Safer Design

Require:

```python
overwrite=True
```

explicitly.

This follows:

```text
Safe by Default
```

One of the most important production engineering principles.

# One Small Improvement We Discovered

Your output became:

```text
2 Alice
3 Bob
1 John
```

instead of:

```text
1 John
2 Alice
3 Bob
```

---

# Why?

Polars:

```python
df.unique()
```

does not guarantee order.

---

# Professional Fix

Use:

```python
df.unique(
    subset=...,
    maintain_order=True,
)
```

---

# Why?

Deterministic outputs are important.

Production systems should produce the same results every run.

---

# Data Lake Concept Introduced

Day 8 introduces an important Data Engineering idea:

```text
Data Lake
```

Typical structure:

```text
data/
├── raw/
├── processed/
└── curated/
```

Example:

```text
raw/jobs.csv
      ↓
transform
      ↓
processed/jobs.parquet
```

Analytics should consume:

```text
processed/
```

not:

```text
raw/
```

---

# Why Keep Raw Data?

Suppose transformation logic changes.

Without raw data:

```text
Need to recollect data
```

With raw data:

```text
Reprocess existing data
```

Very important industry practice.

# Interview Questions You Can Answer

### Why use Parquet instead of CSV?

Parquet is compressed, columnar, and optimized for analytics.

---

### Why keep raw and processed data separately?

To enable reprocessing without recollecting source data.

---

### Why should writes support overwrite protection?

To prevent accidental data loss.

---

### Why are correlation IDs important?

They allow tracing an entire pipeline execution across systems.

---

### Why are deterministic outputs important?

They ensure reproducibility and consistent downstream behavior.
