# Day 7 Notes — Transformation Layer (The "T" in ETL)

Day 7 is one of the most important milestones in the project because this is where raw data finally starts becoming business-ready data.

Until Day 6, our library could:

```text
Read Data
Validate Data
```

But organizations do not derive value from raw data.

They derive value from:

```text
Transformed Data
```

Day 7 solves exactly this problem.

---

# What Was The Goal Of Day 7?

Build a reusable transformation framework for ETL pipelines.

The goal was to convert:

```text
Raw Data
      ↓
Business Data
```

using reusable functions.

Before Day 7:

```python
df = read_file(...)

validate_dataframe(...)
```

After Day 7:

```python
df = read_file(...)

validate_dataframe(...)

df = rename_columns(...)
df = drop_duplicates(...)
df = fill_nulls(...)
df = cast_columns(...)
```

Now we have:

```text
Extract
Validate
Transform
```

which forms the core of almost every Data Engineering pipeline.

---

# Why Do Transformations Exist?

Raw data is rarely useful.

Example:

Input:

```csv
emp_id,name,salary
1,John,50000
2,Alice,70000
```

Business requirement:

```text
Use employee_id instead of emp_id
```

Result:

```csv
employee_id,name,salary
1,John,50000
2,Alice,70000
```

This is transformation.

---

# Real-World Truth

In real ETL systems:

```text
Extract   → 10%
Validate  → 10%
Transform → 70%
Load      → 10%
```

Most engineering effort goes into transformation.

Why?

Because business logic lives here.

---

# Design Philosophy of Day 7

We intentionally avoided creating:

```python
transform_dataframe(...)
```

because transformations are highly business-specific.

Instead we built:

```python
rename_columns()
select_columns()
drop_duplicates()
fill_nulls()
cast_columns()
```

This follows the philosophy of:

```text
Small
Reusable
Composable
Functions
```

---

# Engineering Principle: Composition Over Monoliths

Bad:

```python
transform_everything(df)
```

Huge function.

Difficult to test.

Difficult to debug.

---

Good:

```python
df = rename_columns(df, ...)

df = fill_nulls(df, ...)

df = cast_columns(df, ...)
```

Each function performs exactly one task.

This follows the:

```text
Single Responsibility Principle (SRP)
```

from SOLID principles.

---

# Transformation 1 — Rename Columns

We implemented:

```python
rename_columns()
```

---

## Problem

Different systems use different names.

Example:

System A:

```text
emp_id
```

System B:

```text
employee_id
```

System C:

```text
staff_id
```

---

## Solution

```python
rename_columns(
    df,
    {
        "emp_id": "employee_id"
    }
)
```

---

# Real-World Example

Government migration projects often integrate multiple departments.

One system may use:

```text
emp_code
```

while another uses:

```text
employee_number
```

Standardization becomes mandatory.

---

# Transformation 2 — Select Columns

We implemented:

```python
select_columns()
```

---

## Problem

Input:

```csv
employee_id,name,salary,address,email
```

Analytics only needs:

```text
employee_id
name
salary
```

---

## Solution

```python
select_columns(
    df,
    [
        "employee_id",
        "name",
        "salary",
    ],
)
```

---

# Why This Matters

Benefits:

```text
Less Memory Usage
Faster Queries
Better Security
Reduced Storage
```

---

# Real-World Example

HR data may contain:

```text
Address
Phone Number
PAN Number
```

Analytics team should not receive sensitive information unnecessarily.

---

# Transformation 3 — Drop Duplicates

We implemented:

```python
drop_duplicates()
```

---

## Problem

Input:

```csv
employee_id,name
1,John
1,John
2,Alice
```

Duplicate rows.

---

## Solution

```python
drop_duplicates(
    df,
    subset=["employee_id"],
)
```

---

# Real-World Example

Naukri APIs may return the same job multiple times.

Deduplication prevents:

```text
Duplicate Database Records
Duplicate Notifications
Duplicate Analytics
```

---

# Transformation 5 — Type Casting

We implemented:

```python
cast_columns()
```

---

## Problem

CSV files often load numeric data as strings.

Example:

```text
salary = "50000"
```

instead of:

```text
salary = 50000
```

---

## Solution

```python
cast_columns(
    df,
    {
        "salary": pl.Int64,
    },
)
```

---

# Why This Matters

Operations like:

```python
salary > 50000
```

only work on numeric types.

---

# Real-World Example

Fraud Detection:

```text
transaction_amount
```

must be numeric.

Otherwise:

```python
amount > 10000
```

fails.

# Important Concept: Immutability

Polars transformations return new DataFrames.

Example:

```python
new_df = df.rename(...)
```

instead of modifying:

```python
df
```

directly.

---

# Why Is Immutability Important?

Benefits:

```text
Safer Code
Fewer Bugs
Easier Debugging
Predictable Behavior
```

# Interview Questions You Can Answer

### Why separate validation and transformation?

Validation checks correctness. Transformation changes data into the required business format.

---

### Why use small transformation functions?

Improves reusability, testing, and maintainability.

---

### Why is immutability important?

It prevents accidental modifications and makes debugging easier.

---

### Why use vectorized transformations instead of loops?

Vectorized operations are significantly faster and scale better for large datasets.

---

### What is data standardization?

Converting inconsistent data representations into a common format.
